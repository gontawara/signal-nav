# ADR-002: ルーティングアルゴリズムの選定

- **日付**: 2026-05-13
- **ステータス**: 採用
- **決定者**: gontawara

## 背景

信号最小化ルートを計算するために、ルーティングアルゴリズムとその実装方式を決定する必要がある。既存のルーティングエンジン（OSRM、Valhalla等）のカスタマイズか、自前でのグラフ探索かを選択する。

## 検討した選択肢

### 1. OSRM（Open Source Routing Machine）のカスタムプロファイル

- **概要**: C++製の高速ルーティングエンジン。Luaスクリプトで重みのプロファイルをカスタマイズ可能
- **利点**: 実運用レベルの速度。道路種別ごとの速度設定、ターン制約など高度な機能を標準搭載
- **欠点**: Luaプロファイルの学習コスト、Dockerでのビルド・前処理が必要。前処理に福岡市規模で数十分〜数時間。信号ノードを重みに反映するにはカスタム前処理が必要で、ドキュメントが少ない

### 2. Valhallaのカスタムcosting

- **概要**: C++製ルーティングエンジン。JSONベースのコスティングモデルでカスタマイズ可能
- **利点**: OSRMより柔軟なコスティング。タイル形式でデータを管理するため増分更新が可能
- **欠点**: OSRMと同様にビルド・前処理が必要。Pythonとの統合に追加の工数がかかる

### 3. NetworkX上のDijkstra（信号ペナルティ付き）

- **概要**: OSMnxで取得したNetworkXグラフ上で、エッジの重みを「所要時間＋信号ペナルティ」としてDijkstraを実行する
- **利点**: Pythonのみで完結。OSMnxとの統合がシームレス。重みの調整が容易。デバッグしやすい
- **欠点**: 大規模グラフでは速度が遅い（Pythonのオーバーヘッド）。道路種別ごとの速度設定やターン制約は自前で実装する必要がある

## 決定

**NetworkX上のDijkstra（信号ペナルティ付き）を採用する。**

## 理由

1. **1週間の制約**: OSRM/Valhallaはプロファイルのカスタマイズとビルド環境の構築だけで2〜3日消費する見込み。NetworkXなら初日にプロトタイプが動いた
2. **Pythonで完結**: メイン言語がPythonであり、Luaプロファイルやc++ビルドの学習は1週間に見合わない
3. **デバッグ容易性**: グラフのノード・エッジを直接Pythonで操作できるため、重みの確認やルートの検証が容易
4. **十分な速度**: 福岡市規模（ノード24万、エッジ68万）でDijkstraの計算時間は数秒以内。プロトタイプとして十分

## 実装の詳細

- **重みの計算**: エッジの重み＝距離(m)÷平均速度(m/s)＋信号ペナルティ(秒)
- **信号ペナルティ**: エッジの到達先ノードに信号がある場合に加算。デフォルト60秒
- **平均速度**: 市街地50km/hを仮定。道路種別ごとの速度設定は行わない（スコープ外）
- **重みの方式**: エッジ属性として事前計算し、`nx.shortest_path(G, weight="travel_time")`で属性名を渡す。コールバック関数方式はMultiDiGraphでの挙動が不安定だったため不採用

## リスクと今後の対応

- **速度のスケーラビリティ**: 全国規模に拡張する場合、Pythonでの探索は遅すぎる。その場合はOSRMまたはValhallaへの移行が必要
- **ターン制約の未考慮**: 右左折の信号待ちや、Uターン禁止などは考慮していない。実用化する場合は対応が必要
- **道路種別ごとの速度**: 全道路を一律50km/hとしているが、高速道路や住宅地では速度が異なる。精度向上には道路種別（`highway`タグ）に応じた速度設定が必要
